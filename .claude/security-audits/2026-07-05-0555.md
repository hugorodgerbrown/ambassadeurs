### Low / informational

#### L1 — Token-consuming endpoints are not rate-limited
**Location:** `accounts/views.py:118` (`login_verify`), `public/views.py:382`
(`register_confirm`), `public/views.py:1252` + match action views.
**CWE:** CWE-307 (Improper Restriction of Excessive Authentication Attempts).
**Evidence:** The `POST`/`GET` send-side views (`login_request`,
`register`) carry `@ratelimit` decorators, but the token-*consuming* endpoints
do not. An attacker can submit unlimited candidate tokens.
**Impact:** Effectively none. Tokens are `django.core.signing` values —
HMAC-SHA256 over `SECRET_KEY` with a per-purpose salt. Forging a valid
signature without the key is computationally infeasible, so brute force yields
only 400s. This is a defence-in-depth gap, not an exploitable weakness.
**Recommendation:** Optional. Add a coarse per-IP `@ratelimit` (e.g. `60/h`)
to `login_verify` / `register_confirm` / `match_*` to blunt automated probing
and reduce log noise. Low priority.

#### L2 — Two people can be matched to each other's *own* second registration (self-match via distinct emails)
**Location:** `matching/services.py:165` (`is_eligible_pair`),
`matching/services.py:408` (`propose_match`).
**CWE:** CWE-841 (Improper Enforcement of Behavioral Workflow).
**Evidence:** Eligibility and the candidate query exclude only by
`Registration.pk` (`.exclude(pk=registration.pk)`). Because `Registration` is
`OneToOne` to `User`, one *account* cannot hold both roles — but one *person*
can create two accounts (two emails), register as ambassador on one and referee
on the other, and the engine may pair them. There is no same-person guard
(email domain, phone, or name).
**Impact:** Low and self-limiting. The prior-pass attestation and the actual
discount are verified off-app at the kiosk, where a self-referral cannot
complete (the ambassador must be a genuine prior holder and the referee
genuinely new — the same person cannot be both). A self-match consumes one
ambassador slot until it expires/declines, a minor pool-drain nuisance, not a
PII or integrity breach.
**Recommendation:** Accept as designed (off-app verification is the real gate),
or, if pool-drain abuse is observed, add a soft guard rejecting a proposed pair
whose two registrations share a normalised phone number. Document the decision
in an ADR.

#### L3 — `e2e.yml` pins GitHub Actions by moving tag, not by SHA
**Location:** `.github/workflows/e2e.yml` (`actions/checkout@v5`,
`setup-python@v6`, `astral-sh/setup-uv@v6`, `setup-node@v5`,
`upload-artifact@v4`).
**CWE:** CWE-829 (Inclusion of Functionality from an Untrusted Control Sphere).
**Evidence:** `ci.yml` and `security-audit.yml` pin third-party actions by full
commit SHA; `e2e.yml` uses floating major-version tags, which a compromised or
re-pointed tag could repoint to malicious code.
**Impact:** Low. The e2e workflow runs only on `pull_request` with
`permissions: contents: read` and no secrets beyond ephemeral CI values, so the
blast radius is small.
**Recommendation:** Pin the five actions in `e2e.yml` to commit SHAs, matching
the other two workflows.

#### L4 — npm dev-toolchain vulnerabilities (Lighthouse CI)
**Location:** `package.json` devDependencies → `@lhci/cli@^0.15.1`.
**Evidence:** `npm audit` reports 5 vulns (1 high `tmp` path traversal, 2
moderate `@lhci/cli` / `uuid`, 2 low), all transitive under `@lhci/cli`.
**Impact:** None on the deployed app. `@lhci/cli` is a CI/dev-only tool; it is
never installed on the Render web service (`build.sh` runs `npm ci` then
`npm run css:build`, using only `@tailwindcss/cli`) and none of it ships to the
browser.
**Recommendation:** Bump `@lhci/cli` when a fixed release is available; no
urgency. Consider moving it fully out of the runtime `npm ci` path if not
already excluded on Render.

#### L5 — Stale docstrings claim decline "deletes the decliner's User"
**Location:** `public/views.py:1287-1292` and `:1399-1406`, `_roster_row`
docstring at `public/views.py:1131`.
**Evidence:** Several docstrings/comments state that declining a match deletes
the decliner's `User`/`Registration` and `SET_NULL`s the FK. The current model
(VERB-74 / ADR 0013) *pauses* the registration; FKs are non-nullable `CASCADE`.
**Impact:** Documentation only — no security effect. Flagged because the stale
"deletes user / FK may be NULL" narrative could mislead a future reviewer about
where PII lives after a decline (it is retained, PAUSED, and refundable).
**Recommendation:** Update the docstrings to match the pause-not-delete model.
