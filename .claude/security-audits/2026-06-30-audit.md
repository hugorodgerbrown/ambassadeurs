## Findings by severity

### Medium

#### M1 — Third-party HTMX script loaded without Subresource Integrity (SRI)
**Location:** `templates/base.html:25`
**CWE:** CWE-494 (Download of Code Without Integrity Check)
**Evidence:**
```html
<script src="https://unpkg.com/htmx.org@2.0.4" defer></script>
```
**Impact:** Every page loads executable JavaScript from a third-party CDN with no
`integrity` hash. If unpkg (or its upstream npm tarball) is compromised, or DNS/TLS to
unpkg is subverted, an attacker serves arbitrary script into every authenticated session —
including the match page, where it could exfiltrate revealed contact PII or drive the
accept/decline actions. The version is pinned (`@2.0.4`), which bounds drift but does not
verify content.
**Recommendation:** Either (a) add an SRI hash and `crossorigin="anonymous"`:
`<script src="https://unpkg.com/htmx.org@2.0.4" integrity="sha384-…" crossorigin="anonymous" defer></script>`,
or preferably (b) vendor htmx into `static/` and serve it via WhiteNoise (hashed,
same-origin, offline-resilient) — this also removes a render-blocking third-party request.

#### M2 — No Content-Security-Policy header
**Location:** `config/settings/base.py` / `production.py` (no CSP middleware or `django-csp`)
**CWE:** CWE-693 (Protection Mechanism Failure)
**Evidence:** `MIDDLEWARE` (base.py:43-54) contains no CSP middleware; no `django-csp` in
`INSTALLED_APPS`; no `Content-Security-Policy` header set. The app serves user-supplied
first names and (after accept) email/phone into HTML.
**Impact:** Defence-in-depth gap. Django auto-escaping is the primary XSS control and is
intact (Invariant 4 holds — no `mark_safe`/`|safe` on user data), but with no CSP a single
future escaping mistake, or the un-pinned/un-hashed CDN script (M1), has no second layer to
contain it. A CSP would also constrain the external font/CDN origins already in use.
**Recommendation:** Add a CSP (e.g. `django-csp`) with `default-src 'self'`, an explicit
allow-list for the Google Fonts origins and the htmx source (or self-host per M1), and
`frame-ancestors 'none'`. Start in report-only mode to tune, then enforce.

### Low

#### L1 — Registration form enables account enumeration
**Location:** `matching/forms.py:206-224`
**CWE:** CWE-204 (Observable Response Discrepancy)
**Evidence:** `RegistrationForm.clean()` raises a visible validation error
`_("You are already registered for the current season.")` when a non-UNVERIFIED
registration exists for the submitted email. The login flow (`accounts/views.py:106`)
deliberately does *not* enumerate; this registration path does.
**Impact:** An attacker can probe which email addresses are enrolled in the programme
(membership disclosure). Lower impact than login enumeration since registration is the
public entry point, but it still leaks participation in a privacy-sensitive scheme. The
5/hour per-email rate limit bounds the probing rate.
**Recommendation:** Treat a duplicate like the resend path — accept the submission and
redirect to `register_email_sent` (sending a "you're already registered, here's how to log
in" email to the real owner) rather than surfacing an inline error that distinguishes
registered from unregistered addresses. If the UX requires the inline error, accept the
residual enumeration as a documented trade-off.

#### L2 — GitHub Actions not pinned by commit SHA
**Location:** `.github/workflows/ci.yml`, `security-audit.yml`, `lighthouse.yml`
**CWE:** CWE-829 (Inclusion of Functionality from Untrusted Control Sphere)
**Evidence:** `uses: actions/checkout@v5`, `actions/setup-python@v6`,
`astral-sh/setup-uv@v5`/`@v6`, `actions/setup-node@v4` — all major-version tags, mutable.
**Impact:** A compromised or retagged upstream action could run arbitrary code in CI with
the workflow token. Bounded here because no workflow uses `pull_request_target` and no
deploy secrets are exposed to CI (Render deploys on merge, separately).
**Recommendation:** Pin third-party actions to a full commit SHA with a version comment;
first-party `actions/*` to a SHA is best-practice but lower priority.

#### L3 — Workflows lack an explicit least-privilege `permissions:` block
**Location:** `.github/workflows/*.yml` (no `permissions:` key found)
**CWE:** CWE-272 (Least Privilege Violation)
**Evidence:** `grep permissions .github/workflows/*.yml` → none.
**Impact:** Jobs run with the repository/organisation default `GITHUB_TOKEN` scope, which
may be broader than needed (e.g. `contents: write`). No untrusted-PR trigger is present, so
exposure is limited.
**Recommendation:** Add `permissions: contents: read` at the top of each workflow (raise
specific scopes only where a job needs them).

#### L4 — npm dev-dependency vulnerabilities (Lighthouse CI)
**Location:** `package-lock.json` (`@lhci/cli` → `tmp`, `uuid`, `inquirer`, `external-editor`)
**CWE:** CWE-22 (path traversal in `tmp`), CWE-1284 (bounds check in `uuid`)
**Evidence:** `npm audit` → 1 high (`tmp <=0.2.5`), 2 moderate (`uuid <11.1.1`), 2 low —
all transitive under `@lhci/cli ^0.15.1`, declared in `devDependencies` only.
`build.sh` runs `npm ci` then `npm run css:build` (Tailwind), never `lhci`; Lighthouse runs
only in its own CI workflow.
**Impact:** None on the deployed app — these packages never reach the Render runtime and
the vulnerable code paths (temp-file handling in a CLI) are not exercised by the build.
**Recommendation:** Run `npm audit fix` opportunistically; the `--force` upgrade to
`@lhci/cli@0.1.0` is a breaking downgrade — do **not** apply it. Track upstream `@lhci/cli`
for a non-breaking fix. No launch blocker.

### Info

#### I1 — Self-matching across two accounts is not prevented in-app (accepted by design)
**Location:** `matching/services.py:84-114` (`is_eligible_pair`), `propose_match:165`
**Evidence:** The engine excludes `pk=registration.pk` and the form blocks a second
registration per user/email, but one human registering as an ambassador (email A) and as a
referee (email B) produces two distinct accounts the engine can pair to each other. No
same-person heuristic (e.g. shared phone) exists.
**Impact:** A determined user could match themselves to harvest the referral discount.
Per CLAUDE.md the eligibility proof happens off-app at the kiosk (the ambassador must be a
genuine prior-pass holder and the referee genuinely new), which is the real control;
in-app this is self-attested. Documented as an accepted limitation, not a code defect.
**Recommendation:** No change required for launch. If abuse appears, add a soft signal
(flag matches sharing a phone number / IP for staff review in the admin).

#### I2 — Django admin at the default `/admin/` path
**Location:** `config/urls.py:21`
**Evidence:** `path("admin/", admin.site.urls)`. Admin accounts use passwords (the public
flow is passwordless; staff are the exception).
**Impact:** Default path invites credential-stuffing/scanning. Mitigated by: admin is a
small fixed staff set, `SECURE_SSL_REDIRECT` + secure cookies in production, and Django's
own auth. No public self-serve admin signup.
**Recommendation:** Optional hardening — move admin to a non-guessable path and/or require
strong unique staff passwords (and consider IP-allow-listing via Render). Low priority.

#### I3 — Rate-limit cache is per-process (LocMemCache)
**Location:** `config/settings/base.py:156-160`
**Evidence:** `CACHES.default = LocMemCache`, used by `django-ratelimit`. The code comment
already flags the scale-out caveat.
**Impact:** Correct for the single-instance Render launch. If the web service is ever
scaled to >1 instance, rate-limit counters stop being shared and the login/registration
limits become per-instance (effectively N× looser), weakening the anti-enumeration and
anti-abuse controls. Not an issue at launch topology.
**Recommendation:** Before any horizontal scale-out, switch to a shared backend (Redis).
Keep the existing comment as the tripwire.

## Magic-link / signed-token review (detailed)

The signed-token system (`accounts/tokens.py`) is sound:

- **Single-purpose salts (Invariant 6):** `_LOGIN_SALT` ("accounts.login"),
  `_CONFIRM_SALT` ("accounts.registration-confirm"), `_MATCH_SALT`
  ("accounts.match-access") are all distinct, so a token minted for one action cannot be
  replayed against another (`signing.loads` rejects a mismatched salt).
- **Expiry:** login 1h (`LOGIN_TOKEN_MAX_AGE`), confirm 24h (`MAX_AGE_SECONDS`), match =
  `CONTACT_WINDOW_HOURS * 3600`. All enforced via `signing.loads(..., max_age=...)`.
- **Forgery:** tokens are `TimestampSigner`-signed with `SECRET_KEY`; tampering yields
  `BadSignature` → `None` → 400. `SECRET_KEY` is env-only (`generateValue: true` on Render),
  never in source.
- **Payload minimisation:** tokens carry only PKs (`user_pk` / `registration_pk` /
  `match_pk`), no PII in the URL.
- **Replay / prefetch safety:** `login_verify` GET validates but does **not** log in
  (`accounts/views.py:179`); only POST calls `login()`. Within the 1h window the login
  token is intentionally idempotent (re-submittable) — documented, low risk given the short
  TTL and that re-login produces the same authenticated session.
- **Match token re-derivation:** `_resolve_match_token` (`public/views.py:455`) re-checks
  that the token's `registration_pk` is actually one of the match's two parties before
  granting any view/action — a forged-but-valid token for an unrelated registration is
  rejected.
- **Type confusion:** every reader validates `isinstance(pk, int)` before use.

No token finding raised.

## PII-exposure review (Invariant 1 — the #1 risk)

Verified across views, services, templates, emails, and the account page:

- **Single reveal gate:** `_match_context` adds `counterpart` to the context **only** when
  `match.status == Match.Status.ACCEPTED` (`public/views.py:678-680`). The template renders
  `counterpart.user.email` / `counterpart.phone` **only** inside `{% if counterpart %}`
  (`templates/public/partials/match_actions.html:226,262,277,282,302`). Before ACCEPTED,
  `counterpart` is absent → no PII in the DOM, HTMX swap, or OOB region.
- **Roster / proposed state:** `_roster_row` exposes only first name + initials
  (`public/views.py:599-613`) — consistent with ADR 0009.
- **Account page:** `account_detail` passes only `partner_first_name` (accounts/views.py:280),
  never email/phone.
- **Status polling:** there is no PII-bearing poll endpoint; the match partial re-renders
  state without `counterpart` until ACCEPTED.
- **Emails:** the "matched" notification (`send_match_notification`) carries only a token
  link, no counterpart PII; PII appears only in `send_match_confirmed_email`, called via
  `transaction.on_commit` **after** the match reaches ACCEPTED (`accept_match:485-486`).
- **Declines / expiry:** never reveal PII; decline deletes the decliner's User and SET_NULLs
  the FK; the no-show email to the accused contains no reporter PII.
- **Logging:** services log PKs only — no email/phone at INFO (grep confirmed).

Invariant 1 holds.

## Match integrity / engine review

- **ACCEPTED requires both accepts:** `record_acceptance` sets ACCEPTED only when
  `both_accepted` (both `*_accepted_at` non-null) — `services.py:911-920`. No single-sided
  path reaches ACCEPTED.
- **Eligibility before PROPOSED (Invariant 2):** `propose_match` draws only from
  `eligible_ambassadors()` / `eligible_referees()`, which enforce role, VERIFIED status,
  prior-pass rules, and active-match exclusion; `is_eligible_pair` re-asserts the pair.
  UNVERIFIED registrations never enter matching (`register_participant:649`).
- **1:1 per season (Invariant 3):** `_without_active_match()` excludes any registration
  with a PROPOSED/PENDING/ACCEPTED match from both eligible pools, and `propose_match` runs
  under `select_for_update()` inside the registration transaction — concurrency-safe.
- **No admin/view force-path:** admin marks the match/registration FKs and state fields
  read-only (`matching/admin.py:148-158`); no view transitions a match to ACCEPTED outside
  `record_acceptance`.

## HTMX review

- All mutating fragment endpoints carry `@require_htmx` **and** `@require_POST`
  (`match_accept`, `match_withdraw`, `match_decline`, `match_report_no_show`) — a plain GET,
  even with the HX header, cannot trigger a state change (Invariant 7).
- The only GET partial, `register_details_form`, is `@require_htmx`-guarded and read-only.
- CSRF: middleware active (`CsrfViewMiddleware`); HTMX POSTs go through Django forms with
  the CSRF token. No `csrf_exempt` anywhere (grep clean).
- OOB swap (`register_surface.html:24`, `hx-swap-oob` into `#topband`) targets a trusted,
  server-rendered region with no user data — safe.
- No `HX-Redirect`/`HX-Location`/`HX-Trigger` response headers derived from user input.
